## 🚀 要件定義書：AI特化型ニュースアグリゲーター（仮）

### 1\. プロジェクト概要

| 項目 | 内容 |
| :--- | :--- |
| **プロダクト名（仮）** | Niche Scroller (AI Edition) |
| **プロダクト概要** | 主要なAI関連の情報源（テックブログ、技術記事、論文）を自動集約し、X（旧Twitter）のようなカード形式のタイムラインで「受動的に」閲覧できるWebアプリケーション。 |
| **ターゲットユーザー** | AI分野（特にLLM、機械学習、画像生成など）の最新動向を、効率的にインプットしたい開発者、研究者、学生。 |
| **解決する課題** | 1. **情報源の分散:** AI情報は複数のブログ、記事サイト、論文リポジトリに散らばっており、個別に巡回するのが非効率である。<br>2. **能動的な閲覧の疲労:** 従来のRSSリーダーはメール受信箱型が多く、「記事を探しに行く」能動的な操作が必要。Xのように受動的に情報を流し見したい。 |

### 2\. MVP（最小プロダクト）の定義

#### 2.1. MVPの目的

「**運営が選定したAI情報源を、X風のUIで無限スクロール閲覧できる**」というコア価値を、最小限のコストで実現し、需要を検証する。

#### 2.2. MVPでは実装しない機能（スコープ外）

以下の機能は、需要が確認された後の「フェーズ2」で検討する。

  * ユーザー登録、ログイン、認証機能
  * ユーザーによる情報源の追加・削除・カスタマイズ機能
  * 「いいね」「ブックマーク」「既読/未読」機能
  * 記事のキーワード検索機能
  * ダークモード切り替え（※工数が少なければ実装可）
  * カテゴリ（例：論文、ブログ）によるフィルタリング機能

### 3\. 機能要件

#### 3.1. データ収集要件 (バックエンド / バッチ処理)

1.  **[Must]** 4.2.項で定義された「情報源RSSフィード一覧」を定期的に巡回（クロール）する。（例：1時間に1回）
2.  **[Must]** RSSフィードから新規記事情報を取得する。
3.  **[Must]** 取得した記事がデータベース（DB）に未登録の場合のみ、4.1.項で定義された「収集対象データ項目」をDBに保存する。
4.  **[Should]** 記事の重複登録を防止するため、記事のURLまたはGUIDを一意キーとして管理する。

#### 3.2. データ表示要件 (フロントエンド)

1.  **[Must]** アプリケーションのトップページに、DBに保存された記事情報を時系列（新しい順）で表示する。
2.  **[Must]** 各記事は「カード形式」で表示する。
3.  **[Must]** カードには最低限「記事タイトル」「情報源名（例：OpenAI Blog）」「公開日時（例：3時間前）」を表示する。
4.  **[Should]** OGP画像（記事のサムネイル画像）が取得できている場合は、カード内に画像を表示する。
5.  **[Must]** カード全体をクリック（タップ）すると、ブラウザの新しいタブで元記事のURLを開く。
6.  **[Must]** ページの最下部までスクロールすると、自動で次の記事データを読み込み、表示する。（＝無限スクロール）

### 4\. データ要件

#### 4.1. 収集対象データ項目 (DBスキーマ案)

データベースの `articles` テーブル（仮）に保存するカラム定義。

| カラム名（案） | データ型 | 説明 | 必須 |
| :--- | :--- | :--- | :--- |
| `id` | UUID | 一意な記事ID | **Must** |
| `title` | TEXT | 記事タイトル | **Must** |
| `article_url` | TEXT | 元記事のURL (UNIQUE) | **Must** |
| `published_at` | TIMESTAMP | 公開日時 | **Must** |
| `source_name` | TEXT | 情報源名（例：'Zenn'） | **Must** |
| `summary` | TEXT | 記事の概要（RSSにあれば） | Should |
| `image_url` | TEXT | OGP画像のURL（RSSにあれば） | Should |
| `created_at` | TIMESTAMP | DBへの登録日時 | **Must** |

### 5\. 非機能要件

#### 5.1. 運用・インフラ

  * **[Must]** フロントエンド、バックエンド（API）、DB、バッチ処理のすべてが、\*\*無料枠（Free Tier）\*\*の範囲内で運用可能であること。
  * **[Must]** デプロイ作業が容易であること。（例：Gitプッシュによる自動デプロイ）

#### 5.2. パフォーマンス

  * **[Should]** タイムラインの初期表示（LCP）が高速（例：3秒以内）であること。

### 6\. （参考）想定アーキテクチャ

  * **フロントエンド / ホスティング:** Next.js (App Router) / Vercel
  * **データベース:** Supabase (PostgreSQL)
  * **情報収集バッチ:** GitHub Actions (cronによるスケジュール実行)
  * **言語:** TypeScript (Next.js), Python or Node.js (バッチ)
